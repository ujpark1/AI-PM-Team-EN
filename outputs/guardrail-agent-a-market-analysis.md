## 1. 시장 분석

### TAM / SAM / SOM 추정

| 구분 | 추정 규모 | 근거 |
|------|-----------|------|
| **TAM** | ~$12B | 글로벌 애플리케이션 보안 시장 (2025-2026 기준). SAST, DAST, RASP, SCA 전체 포함 |
| **SAM** | ~$800M - $1.2B | "비개발자/로우코드/노코드 빌더"가 만드는 앱의 보안 수요. 전체 앱보안 시장의 약 7-10% |
| **SOM** | ~$5M - $15M (3년 내) | 영어권 + 한국 시장에서 "AI 코딩 도구로 앱을 만드는 비개발자" 대상 |

**솔직한 평가:** TAM은 충분히 크고, SAM은 빠르게 성장 중. PRD의 12개월 목표 MRR $15,000(연간 $180K)은 SOM 대비 매우 보수적 — 이것은 좋은 신호. 과대망상이 아닌 실행 가능한 목표. 다만 VC 투자를 받으려면 이 숫자로는 설득이 어렵고, 부트스트래핑이나 시드 라운드 수준.

### 트렌드 분석: 바이브코딩 시장의 현재와 미래

**현재 상황 (2026년 2월 기준):**

바이브코딩은 "트렌드"를 넘어 **구조적 변화**의 초기 단계.

- **공급 측면:** Claude Code, Cursor, Replit, Bolt, v0 등이 폭발적으로 성장. AI 코딩 도구의 성능이 "장난감"에서 "프로덕션급"으로 넘어가는 변곡점을 지남
- **수요 측면:** "나는 개발자가 아닌데 앱을 만들었다"는 사례가 주 단위로 바이럴. 마케터, 디자이너, PM이 직접 제품을 출시하는 현상이 가속화
- **핵심 인사이트:** 이들이 만든 앱은 "프로토타입"이 아니라 **실제로 결제를 받는 프로덕션 서비스**. 여기서 보안 갭이 발생

**미래 전망 (12-36개월):**

- 바이브코더 인구는 현재 100만 명 수준에서 3년 내 500만-1,000만 명으로 성장 가능
- AI 코딩 도구의 보안 내장 기능이 발전하겠지만, "충분히 좋은" 수준에 도달하려면 최소 2-3년 소요
- 이 2-3년의 윈도우가 Guardrail의 기회

### 페인포인트 매핑

```
[심각도: 높음] ─── "내 앱이 해킹당하면 어떡하지?" (막연한 공포)
       │
       ├── "보안이 뭔지는 아는데, 어디서부터 시작할지 모르겠다"
       ├── "Snyk 가입했는데 대시보드가 무슨 말인지 모르겠다"
       ├── "보안 컨설턴트 비용이 시간당 30만원이다"
       └── "Product Hunt 출시인데, 혹시 개인정보가 새고 있으면..."

[심각도: 중간] ─── "사용자한테 '안전하다'고 말해야 하는데 근거가 없다"
       │
       ├── "SOC2? ISO27001? 그게 뭔데?"
       └── "B2B 고객이 보안 감사 결과를 요구한다"

[심각도: 낮음] ─── "나중에 개발자 고용하면 해결되겠지"
                    (→ 이 층은 전환이 어려움)
```

**핵심 발견:** 심각도 "높음" 영역의 페인포인트가 매우 구체적이고 감정적. 사람들은 논리가 아니라 **공포**로 지갑을 엶.

### 타이밍 판단: "왜 지금인가?"

**10점 만점에 9점 — 타이밍이 거의 완벽.**

| 타이밍 요소 | 판단 |
|------------|------|
| 기술 성숙도 | AI 코딩 도구가 "프로덕션급 앱 생성" 수준에 도달한 지 12-18개월 |
| 사용자 인구 | 비개발자 앱 빌더가 임계 질량(critical mass)에 도달하는 시점 |
| 사고 발생 | 아직 "바이브코딩 앱에서 대규모 데이터 유출" 뉴스가 터지지 않았음. 터지면 수요 폭증 |
| 경쟁 부재 | 이 세그먼트를 정면으로 노리는 플레이어가 사실상 없음 |
| 플랫폼 지원 | Vercel, Netlify 등이 원클릭 배포를 지원하여 비개발자의 프로덕션 배포가 일상화 |

**엑싯 경험에서 배운 교훈:** "대형 사고가 터지기 직전"에 포지셔닝되어 있는 것이 가장 좋은 타이밍. 바이브코딩 보안 사고는 시간 문제. 그때 이미 시장에 있어야 함.

## 2. 기회 패턴 인식

### 해당하는 기회 패턴

**패턴 1: 기술의 민주화 (Primary)**
- AI 코딩 도구가 "코딩 능력"을 민주화 → 새로운 사용자층(비개발자) 탄생 → 이들을 위한 보안도 민주화 필요
- "민주화의 2차 파동" — 1차 파동(AI 코딩)이 만든 새로운 시장의 인접 수요

**패턴 2: 행동 변화의 파도 타기 (Secondary)**
- "비개발자가 앱을 만들어 출시한다"는 행동 자체가 2023-2026년에 새로 생긴 행동

**패턴 3: 기존 산업의 소프트웨어화 (Tertiary)**
- 보안 감사/컨설팅이라는 전통적 "서비스"를 셀프서비스 SaaS로 변환

### Blue Ocean 4 Actions Framework 적용

```
┌─────────────────────────────────────────────────────┐
│                Blue Ocean 4 Actions                  │
├──────────────┬──────────────────────────────────────┤
│  제거        │ • 전문 보안 용어 (OWASP, CVE 등)     │
│ (Eliminate)  │ • CLI 의존성                         │
│              │ • 복잡한 온보딩 프로세스               │
│              │ • 개발자 팀 협업 기능 (초기)           │
├──────────────┼──────────────────────────────────────┤
│  감소        │ • 스캔 깊이 (100% → 핵심 80%)        │
│ (Reduce)     │ • 설정 옵션 수                        │
│              │ • 리포트 복잡성                       │
├──────────────┼──────────────────────────────────────┤
│  증가        │ • 설명의 친근함과 교육적 가치         │
│ (Raise)      │ • 감정적 안심감 (뱃지, 점수)          │
│              │ • 온보딩 속도 (30초 이내 첫 결과)     │
├──────────────┼──────────────────────────────────────┤
│  창조        │ • "건강검진" 메타포 전체              │
│ (Create)     │ • 비개발자 친화적 보안 대시보드       │
│              │ • "축하의 언어" (보안 = 성취)          │
│              │ • MCP 통합 (AI 코딩 도구 네이티브)    │
│              │ • URL만 붙여넣기 스캔                 │
└──────────────┴──────────────────────────────────────┘
```

**전략적 평가:** Guardrail이 기존 보안 도구와 **같은 차원에서 경쟁하지 않는다**는 점이 핵심. "더 나은 Snyk"이 아니라 "완전히 다른 카테고리". 카테고리 창조는 성공하면 크게, 실패하면 "시장이 없었다"로 끝남.

## 3. JTBD (Jobs To Be Done) 분석

### Functional Job (기능적 과업)

> **"내가 AI로 만든 앱이 보안 취약점 없이 안전하게 운영되고 있는지 확인하고 싶다"**

### Emotional Job (감정적 과업)

> **"내 앱을 세상에 내놓을 때, '보안 때문에 큰일 날 수도 있다'는 불안감에서 해방되고 싶다"**

이것이 **진짜 Job**. Alex는 보안 전문가가 되고 싶은 게 아니라, **불안하지 않고 싶을 뿐**.

### Social Job (사회적 과업)

> **"나는 비개발자지만, 보안까지 신경 쓰는 프로페셔널한 창업자로 인정받고 싶다"**

**전략적 시사점:** PRD가 Emotional Job을 매우 잘 이해하고 있음. 다만 **Social Job에 대한 설계가 아직 약함** — 뱃지, 공유 기능, 커뮤니티 요소가 로드맵에 더 명시적으로 들어가야 함.

## 5. 포지셔닝 & 차별화 평가

### 포지셔닝

```
        단순 & 저렴
            │
            │  ★ Guardrail (비어있는 공간)
            │
비개발자 ───┼─── 개발자
친화적      │      친화적
            │
            │  Snyk, Semgrep, GitHub Security
            │

```

**포지셔닝은 정확하지만 두 가지 주의점:**

1. **"비어있는 이유"**: 이전에는 비개발자가 프로덕션 앱을 만드는 현상 자체가 없었으므로 → **(a) 기회**에 가깝다고 판단
2. **방어 가능성**: Snyk/GitHub가 "Easy Mode"를 출시하면? 방어는 **브랜드와 커뮤니티**로 해야 함


## 7. 흥분 지수: 8.0 / 10
