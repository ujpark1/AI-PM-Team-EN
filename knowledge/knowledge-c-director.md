# Knowledge Base — Agent C (프로덕트 디렉터)

## 1. 의사결정 프레임워크

### RAPID 의사결정 모델
- **R (Recommend)**: 제안자 — A와 B가 분석과 추천을 제공
- **A (Agree)**: 동의자 — 핵심 이해관계자의 동의 필요 여부
- **P (Perform)**: 실행자 — 결정 후 실행을 담당할 주체
- **I (Input)**: 입력 제공자 — 의사결정에 필요한 정보를 제공하는 사람
- **D (Decide)**: 결정자 — **C(디렉터)가 최종 결정**

### 의사결정 속도 vs 품질 매트릭스
| | 되돌리기 쉬움 | 되돌리기 어려움 |
|--|--------------|----------------|
| **영향도 높음** | 빠르게 결정, 바로 실행 | 신중하게 결정, 충분한 검토 |
| **영향도 낮음** | 즉시 결정, 위임 가능 | 빠르게 결정, 모니터링 |

**Jeff Bezos의 Type 1 / Type 2 결정**:
- **Type 1 (일방문)**: 되돌리기 어려운 결정 → 신중하게, 데이터 기반으로
- **Type 2 (양방문)**: 되돌리기 쉬운 결정 → 빠르게, 70% 확신이면 GO

### 의사결정 안티패턴 (피해야 할 것)
1. **Analysis Paralysis**: 완벽한 데이터를 기다리며 결정을 미루는 것
2. **HiPPO**: Highest Paid Person's Opinion — 직급이 높은 사람의 직관에만 의존
3. **Sunk Cost Fallacy**: "이미 투자했으니 계속 가자" — 미래 가치로 판단해야
4. **Confirmation Bias**: 원하는 결론을 지지하는 데이터만 찾는 것
5. **Groupthink**: 모두가 동의하면 오히려 의심하라

## 2. 제품 전략 프레임워크

### Product Vision → Strategy → Roadmap
- **Vision**: 3-5년 후 세상이 어떻게 변해 있을 것인가?
- **Strategy**: 그 비전을 향해 어떤 경로로 갈 것인가?
- **Roadmap**: 다음 분기/반기에 무엇을 할 것인가?
- **핵심 노하우**: 비전은 고정, 전략은 연간, 로드맵은 분기별 업데이트

### RICE 우선순위 프레임워크
- **R (Reach)**: 이 기능이 영향을 미치는 사용자 수
- **I (Impact)**: 개별 사용자에 대한 영향 (0.25/0.5/1/2/3)
- **C (Confidence)**: 추정의 확신도 (%)
- **E (Effort)**: 필요한 작업량 (person-weeks)
- **RICE Score = (Reach × Impact × Confidence) / Effort**

### MoSCoW 우선순위
- **Must have**: 없으면 제품이 의미 없음 (MVP 필수)
- **Should have**: 중요하지만 없어도 동작함 (v1.1)
- **Could have**: 있으면 좋지만 우선순위는 낮음 (backlog)
- **Won't have (this time)**: 이번에는 안 함 (명시적 제외)

## 3. Go-To-Market (GTM) 전략 노하우

### GTM 모션 유형

**1. Product-Led Growth (PLG)**
- 제품 자체가 획득과 전환을 주도
- 적합: 셀프서비스 가능, 낮은 진입 장벽, 바이럴 요소 존재
- 예: Slack, Notion, Figma, Dropbox
- **핵심 지표**: 가입 → 활성화율, 무료 → 유료 전환율, 바이럴 계수(k)
- **노하우**: PLG에서 Aha Moment까지의 시간을 최소화하는 것이 핵심

**2. Sales-Led Growth (SLG)**
- 영업팀이 고객 확보를 주도
- 적합: B2B, 높은 계약 가치(ACV), 복잡한 의사결정 구조
- **핵심 지표**: 파이프라인 가치, 전환율, 평균 세일즈 사이클, ACV
- **노하우**: 초기에는 창업자 세일즈(Founder-led Sales)가 가장 효과적

**3. Community-Led Growth (CLG)**
- 커뮤니티가 인지도와 신뢰를 구축
- 적합: 개발자 도구, 오픈소스, 크리에이터 도구
- **핵심 지표**: 커뮤니티 크기, 기여자 수, 콘텐츠 생산량
- **노하우**: 커뮤니티는 가꾸는 것이지 만드는 것이 아님. 진정성이 핵심.

### 런칭 전략

**Soft Launch (소프트 론칭)**
- 소규모 그룹 대상 조용한 출시
- 목적: 버그 발견, 초기 피드백, 핵심 지표 베이스라인 확보
- 기간: 2-4주

**Public Launch (퍼블릭 론칭)**
- Product Hunt, Hacker News, 소셜 미디어, 프레스 등
- **Product Hunt 노하우**:
  - 화요일-목요일 PST 자정에 게시
  - 커뮤니티에 미리 공유 (하지만 upvote 요청은 No)
  - 첫 1시간이 가장 중요
  - 헌터(Hunter) 섭외가 유리

**단계적 론칭 (Phased)**
1. **Alpha**: 내부 테스트 + 친한 사용자 5-10명
2. **Beta (Closed)**: 초대제로 50-200명
3. **Beta (Open)**: 누구나 가입 가능, 대기열 활용
4. **GA (General Availability)**: 정식 출시

### 프라이싱 전략
- **침투 가격**: 낮은 가격으로 빠르게 시장 점유 → 이후 인상
- **가치 기반 가격**: 고객이 얻는 가치의 10-20%
- **경쟁자 기반**: 경쟁자 대비 10-20% 낮거나, 프리미엄 포지셔닝
- **핵심 노하우**: 가격은 올리기보다 내리기가 훨씬 어렵다. 처음부터 너무 싸게 가지 마라.

## 4. 출시 후 성과 관리 노하우

### 30-60-90일 프레임워크

**Day 1-30: Validate (검증)**
- 목표: 핵심 가설이 맞는지 확인
- 집중 지표: 가입률, 활성화율, 초기 리텐션
- 의사결정: 핵심 기능이 제대로 작동하는가?
- **B에게 요청할 분석**: 가입 퍼널 분석, 활성화 행동 분석

**Day 31-60: Optimize (최적화)**
- 목표: 약한 지표를 개선
- 집중 지표: 전환율, 리텐션, NPS
- 의사결정: 어디에 리소스를 집중할 것인가?
- **B에게 요청할 분석**: 코호트 분석, 이탈 원인 분석, A/B 테스트 결과

**Day 61-90: Scale or Pivot (확장 또는 전환)**
- 목표: 데이터 기반 방향 결정
- 집중 지표: LTV/CAC, 유닛 이코노믹스, 성장률
- 의사결정: 스케일 / 피벗 / 킬
- **B에게 요청할 분석**: 세그먼트 분석, 성장 모델링, 시나리오 분석

### 피벗 판단 기준

**피벗이 필요한 신호**:
- 리텐션이 평탄해지지 않고 계속 하락 (leaky bucket)
- 핵심 가설이 데이터로 부정됨
- CAC가 지속적으로 상승하고 LTV가 개선되지 않음
- 파워 유저가 우리가 예상한 것과 다른 방식으로 제품을 사용

**피벗 유형**:
1. **Zoom-in Pivot**: 하나의 기능을 전체 제품으로 확장
2. **Zoom-out Pivot**: 전체 제품을 더 큰 제품의 하나의 기능으로
3. **Customer Segment Pivot**: 같은 제품, 다른 타겟
4. **Value Capture Pivot**: 같은 제품, 다른 수익 모델
5. **Channel Pivot**: 같은 제품, 다른 유통 채널
6. **Technology Pivot**: 같은 문제, 다른 기술 솔루션

## 5. 이해관계자 커뮤니케이션

### SCQA 프레임워크 (상황-복잡-질문-답변)
- **S (Situation)**: 현재 상황/배경
- **C (Complication)**: 문제/도전
- **Q (Question)**: 해결해야 할 핵심 질문
- **A (Answer)**: 제안하는 해결책

### 의사결정 문서 작성 템플릿
```
## 제목: [결정 사항]
## 날짜: [YYYY-MM-DD]
## 상태: Draft / Under Review / Approved

### 배경
왜 이 결정이 필요한가?

### 옵션
| 옵션 | 장점 | 단점 | 리스크 |
|------|------|------|--------|
| 옵션 A | ... | ... | ... |
| 옵션 B | ... | ... | ... |
| 옵션 C | ... | ... | ... |

### 결정
선택한 옵션과 근거

### 번복 조건
어떤 데이터가 나오면 이 결정을 재검토할 것인가?

### 다음 단계
누가, 무엇을, 언제까지
```

## 6. 제품 성장 엔진

### 세 가지 성장 엔진 (Eric Ries)

**1. Sticky Engine (점착 엔진)**
- 핵심: 높은 리텐션 → 새 사용자 > 이탈 사용자 → 성장
- 핵심 지표: 이탈률(Churn Rate)
- 전략: 제품 핵심 가치 강화, 습관 형성, 전환 비용 높이기
- 적합: 유틸리티, 생산성 도구, 인프라

**2. Viral Engine (바이럴 엔진)**
- 핵심: 사용자가 사용자를 데려옴 → 바이럴 계수(k) > 1
- 핵심 지표: 바이럴 계수(k), 바이럴 사이클 타임
- 전략: 공유 인센티브, 네트워크 효과, 소셜 기능
- 적합: 소셜, 커뮤니케이션, 협업 도구

**3. Paid Engine (유료 엔진)**
- 핵심: LTV > CAC → 투자한 만큼 성장
- 핵심 지표: LTV/CAC 비율, 페이백 기간
- 전략: 마케팅 채널 최적화, 세일즈 효율화
- 적합: B2B SaaS, 높은 ACV 제품

### North Star Metric 연결
C는 성장 엔진에 맞는 North Star Metric을 선정하고, B에게 이를 중심으로 데이터 추적 및 분석을 요청해야 합니다.

| 성장 엔진 | North Star Metric 방향 |
|-----------|----------------------|
| Sticky | WAU/MAU 비율, D7/D30 리텐션 |
| Viral | 초대/공유 수, 바이럴 계수(k) |
| Paid | LTV/CAC 비율, Payback Period |
